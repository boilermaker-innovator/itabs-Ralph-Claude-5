What this is: Ralph Loop is a technique that keeps Claude Code running in a loop - it keeps trying until your task is actually done. Named after Ralph Wiggum from The Simpsons because it "stubbornly keeps iterating, failing repeatedly but never gives up." Ralph TUI adds a visual terminal interface to manage it all.

1 What Is Ralph Loop? CONCEPT

Ralph Loop in Plain English

Normally when you give Claude Code a task, it does one pass and stops - even if it didn't fully finish. Ralph Loop changes that.

The idea: When Claude tries to exit, Ralph intercepts it and says "nope, try again" - feeding the same prompt back. Claude sees the files it just changed, so each loop builds on the last.

It keeps going until either:

  • It meets your success criteria (you define what "done" looks like)
  • It hits your max iteration limit (your safety net)
Give Task ‚Üí Claude Works ‚Üí Tries to Exit ‚Üí Ralph Blocks ‚Üí Loop Again

The Technical Bit

At its core, Ralph is literally just a bash loop. Geoffrey Huntley (who invented this) described it simply:

# The original Ralph - dead simple
```

while true; do
claude ‚ÄúYour prompt here‚Äù
done

The official Claude Code plugin wraps this with better controls:

  • Stop hook - Intercepts Claude‚Äôs exit and re-feeds the prompt
  • Completion detection - Looks for your success token like COMPLETE
  • Max iterations - Safety limit so it doesn‚Äôt run forever
  • Session persistence - Can pause/resume without losing progress

When Ralph Loop Shines

Best for mechanical, well-defined tasks:

  • Large refactors - Framework migrations, dependency upgrades across hundreds of files
  • Test coverage - ‚ÄúAdd tests until we hit 80% coverage‚Äù
  • Batch operations - Update every API endpoint to new format
  • Bug hunting - Keep fixing until all tests pass
  • Documentation - Document every function in codebase

NOT good for:

  • Tasks requiring human judgment at each step
  • Creative work where direction might change
  • Anything where you can‚Äôt define clear ‚Äúdone‚Äù criteria
üí° The Philosophy

‚ÄúBetter to fail predictably than succeed unpredictably.‚Äù - Geoffrey Huntley. Each failure teaches Claude something. The skill shifts from directing step-by-step to writing prompts that converge toward correct solutions.

Quick Decision Guide

Use Ralph Loop (plugin) when:

  • You have ONE specific task to loop on
  • Task is well-defined with clear completion criteria
  • You don't need to track multiple sub-tasks
  • Example: "Migrate all Jest tests to Vitest"

Use Ralph TUI when:

  • You're building something with multiple features
  • You want to see real-time progress visually
  • You need to pause/resume complex work
  • You want to run multiple agents at once
  • Example: "Build a complete second-brain app"
üí° Starting Point

If you're new to this, start with the Ralph Loop plugin on a small task. Get comfortable with autonomous loops before jumping to Ralph TUI's multi-agent orchestration.

4 Ralph TUI Setup FULL SETUP

What You Need First

  1. Bun - JavaScript runtime (like Node but faster) 
    curl -fsSL https://bun.sh/install | bash
  2. Claude Code CLI - Or OpenCode as alternative 
    curl -fsSL https://claude.ai/install.sh | bash
  3. A project to work on - Existing codebase or new folder

Installation Steps

1. Install Ralph TUI globally:

bun install -g ralph-tui

2. Navigate to your project:

cd your-project

3. Run the setup wizard:

ralph-tui setup

4. Answer the wizard questions:

  • Issue tracker: Choose JSON file (simplest) - tracks tasks in local prd.json
  • AI agent: Choose Claude Code (or OpenCode)
  • Max iterations: Set to 35 (safe starting point)
  • Auto-complete: Keep on for task completion

Creating Your PRD (Product Requirements Doc)

The PRD is an AI conversation where you describe what you want to build. Ralph uses this to create a task list.

Start the PRD chat:

ralph-tui create-prd --chat

The AI will ask you questions like:

  • What are you building?
  • What's the platform? (web, mobile, etc.)
  • What features do you need?
  • What's your MVP scope? (quick prototype vs full build)

Be as descriptive as possible. The more detail you give, the better the task breakdown.

After PRD is done:

  • It generates a detailed PRD document
  • You can review and tweak it
  • Then create a task list from it
üí° Scope Warning

If you say "8-12 week project", Ralph will work rigorously on every feature and burn tokens like crazy. For testing, choose "quick prototype" scope first.

Running Ralph TUI

Start the TUI:

ralph-tui run --prd ./prd.json

Keyboard controls in the TUI:

  • S - Start/deploy agents on tasks
  • T - Toggle trace view (see what agents are doing)
  • P - Pause the loop
  • Q - Quit

What you'll see:

  • Left panel: Task list with status (pending, in-progress, done)
  • Right panel: Live agent output
  • Bottom: Controls and stats

Running multiple agents:

You can start different tasks simultaneously. Press S on each task you want to run in parallel.

Hard-Won Lessons

Keep change sets manageable:

For existing codebases, set Ralph to run ONCE overnight on a cron, making small iterations. Waking up to one small refactor daily beats waking up to 50 massive changes.

Define "done" clearly:

Vague prompts = infinite loops. "Make it better" will run forever. "All tests passing with >80% coverage" has a clear endpoint.

Start conservative with iterations:

  • Small task: 10-20 iterations
  • Medium task: 30-50 iterations
  • Complex project: 100+ (monitor closely!)

Autonomous ‚â† unsupervised:

Check in periodically. Ralph might get stuck in a loop or go in the wrong direction. Human judgment still matters for strategic decisions.

üí° The Real Skill

Ralph doesn't replace human judgment - it automates mechanical execution. The skill shifts from "directing Claude step by step" to "writing prompts that converge toward correct solutions."
